# 100 Viva Questions and Answers for Compiler Design Case Study

## Section 1: Intermediate Code Generation (TAC)
1. **What is Intermediate Code?**
   Intermediate code is a machine-independent representation of the source program generated by the compiler after syntax and semantic analysis.
2. **Why do compilers generate intermediate code?**
   It decouples the front-end (language-specific) from the back-end (machine-specific), enabling retargetability and machine-independent optimizations.
3. **What is Three Address Code (TAC)?**
   TAC is an intermediate representation where each instruction has at most one operator on the right side and at most three addresses (operands/result).
4. **What are Quadruples?**
   A data structure for TAC containing four fields: Operator, Argument 1, Argument 2, and Result.
5. **What are Triples?**
   A data structure for TAC containing three fields: Operator, Argument 1, and Argument 2. The result is referred to by its position/index.
6. **What is the advantage of Quadruples over Triples?**
   Quadruples allow easy code movement during optimization because results are explicitly named, whereas Triples rely on position indices that break if code is moved.
7. **What is Indirect Triples representation?**
   It uses an array of pointers to Triples, making it easier to optimize code by simply moving pointers rather than modifying the triples themselves.
8. **How does TAC handle arrays?**
   Using special instructions like `t1 = a[i]` and `a[j] = t2`.
9. **How do we generate TAC for `x = a + b * c`?**
   `t1 = b * c`, `t2 = a + t1`, `x = t2`.
10. **Why are temporary variables used in TAC?**
    To hold the intermediate results of complex expressions so that each instruction strictly adheres to the three-address format.

## Section 2: Boolean Expressions & Short Circuit Evaluation
11. **How are boolean expressions translated in compilers?**
    They can be translated either numerically (1 for true, 0 for false) or using control flow (branching).
12. **What is short-circuit evaluation?**
    Evaluating boolean expressions only until the truth value is definitively known. E.g., in `A or B`, if A is true, B is not evaluated.
13. **Why is short-circuit evaluation important?**
    It saves execution time and prevents runtime errors (e.g., `if (ptr != NULL && ptr->val == 1)`).
14. **How do we generate TAC for `if (A < B and C < D)`?**
    `if A < B goto L1; goto Lfalse; L1: if C < D goto Ltrue; goto Lfalse;`
15. **What is a branch target?**
    The label or instruction index to which control is transferred during a jump.
16. **How does the `or` operator work in control flow translation?**
    If the first operand is true, jump directly to the true exit. If false, fall through to evaluate the second operand.
17. **How does the `not` operator affect jump targets?**
    It swaps the true and false exit targets of the expression.
18. **Why don't we generate numerical TAC for `if` conditions?**
    Because `if` conditions strictly dictate flow of control, so translating directly to jumps is more efficient.
19. **How does YACC/Bison evaluate boolean precedence?**
    By assigning `%left` and `%right` precedence rules in the grammar definition.
20. **What happens if a short-circuit expression is part of an assignment (`x = a < b or c < d`)?**
    Control flow must branch to blocks that explicitly assign 1 to `x` or 0 to `x`.

## Section 3: Backpatching
21. **What is backpatching?**
    A technique to resolve forward branches in a single pass compiler by leaving jump targets blank and filling them in later when the target is known.
22. **Why is backpatching used?**
    Because when a forward jump is generated (like the end of a `while` loop or `if` statement), the destination instruction hasn't been parsed yet.
23. **What is a `truelist`?**
    A list of instruction indices that should jump to the location executed when the boolean expression is true.
24. **What is a `falselist`?**
    A list of instruction indices that should jump to the location executed when the boolean expression is false.
25. **What is a `nextlist`?**
    A list of instruction indices (usually unconditional jumps) that should jump to the statement immediately following the current construct.
26. **What does the `makelist(i)` function do?**
    It creates a new list containing only the instruction index `i`.
27. **What does the `merge(p1, p2)` function do?**
    It concatenates two lists `p1` and `p2` to form a single list.
28. **What does the `backpatch(p, i)` function do?**
    It traverses the list `p` and fills the target of those instructions with the index `i`.
29. **How are labels generated dynamically?**
    By maintaining a global counter and incrementing it (e.g., `L1`, `L2`) or by tracking instruction indices directly in an array/linked list.
30. **In an `if-else` statement, where does the `goto` at the end of the `if` block point?**
    It points to the statement immediately following the entire `if-else` construct, escaping the `else` block.

## Section 4: Basic Block Formation
31. **What is a Basic Block?**
    A sequence of consecutive instructions with exactly one entry point and one exit point.
32. **What does "single entry" mean?**
    Control can only enter the block at its first instruction; there are no jumps into the middle of the block.
33. **What does "single exit" mean?**
    Control leaves the block only at its last instruction; there are no jumps outward from the middle.
34. **What is a Leader?**
    The first instruction of a Basic Block.
35. **What is the first rule of Leader Identification?**
    The very first instruction of the intermediate code is a leader.
36. **What is the second rule of Leader Identification?**
    Any instruction that is the target of a conditional or unconditional jump is a leader.
37. **What is the third rule of Leader Identification?**
    Any instruction that immediately follows a conditional or unconditional jump is a leader.
38. **Why do we identify leaders?**
    To partition the intermediate code sequence into disjoint Basic Blocks.
39. **How do you find the end of a Basic Block?**
    A Basic Block ends at the instruction just before the next leader, or at the end of the program.
40. **Can a Basic Block contain a `goto` instruction?**
    Yes, but it must be the very last instruction of the block.
41. **Can a Basic Block contain a label?**
    Yes, but it must be the very first instruction of the block (making it a leader).
42. **What data structure is typically used to store a Basic Block?**
    A linked list of instructions or an array of instruction indices, along with a block ID.
43. **Why are Basic Blocks critical for compilers?**
    They are the fundamental unit for local optimizations and Control Flow Graph construction.
44. **What happens if an instruction is never reached?**
    It forms an unreachable Basic Block (Dead Code) and can be eliminated.
45. **Does a function call end a Basic Block?**
    Usually yes, because control is transferred out of the current scope, though it depends on compiler implementation.

## Section 5: Control Flow Graphs (CFG)
46. **What is a Control Flow Graph?**
    A directed graph where nodes represent Basic Blocks and edges represent control flow paths between them.
47. **What constitutes an edge in a CFG?**
    A directed edge from B1 to B2 exists if control can transfer from the end of B1 to the start of B2.
48. **When does a "fall-through" edge occur?**
    When B2 immediately follows B1 in sequence, and B1 does not end with an unconditional jump.
49. **What are successors?**
    The blocks to which control can flow directly from the current block.
50. **What are predecessors?**
    The blocks from which control can flow directly into the current block.
51. **Why is the CFG important?**
    It provides the structural representation of the program needed for global data-flow analysis and loop optimizations.
52. **How does a `while` loop appear in a CFG?**
    As a cycle or back-edge in the graph.
53. **How does an `if-else` statement appear in a CFG?**
    A block splitting into two paths (true and false), which later merge at a common successor block.
54. **What is a DAG (Directed Acyclic Graph)?**
    A graph with no cycles. A CFG without loops is a DAG. DAGs are also used inside basic blocks for expression evaluation.
55. **How is a CFG represented in code?**
    Using an Adjacency Matrix or Adjacency List.
56. **What is Graphviz DOT?**
    A graph description language used to visually render nodes and edges, often used for visualizing CFGs.
57. **What is an Entry Node?**
    A unique node added to the CFG representing the start of the program.
58. **What is an Exit Node?**
    A unique node representing the termination of the program.
59. **What is a dominator in a CFG?**
    Node A dominates Node B if every path from the entry node to B must pass through A.
60. **Why do we analyze dominators?**
    To identify natural loops for loop invariant code motion.

## Section 6: Local Code Optimization
61. **What is Code Optimization?**
    The process of modifying code to make it run faster or consume less memory without changing its output.
62. **What is Local Optimization?**
    Optimizations performed strictly within a single Basic Block.
63. **What is Global Optimization?**
    Optimizations performed across Basic Blocks using data-flow analysis on the CFG.
64. **What is Constant Folding?**
    Evaluating constant expressions at compile time (e.g., replacing `3 * 4` with `12`).
65. **What is Constant Propagation?**
    Replacing variables with their known constant values if they haven't been modified.
66. **What is Copy Propagation?**
    If `x = y`, replacing subsequent uses of `x` with `y` to eliminate the unnecessary copy.
67. **What is Common Subexpression Elimination (CSE)?**
    Identifying if an expression like `a + b` was already computed, and replacing re-computations with the previously stored result.
68. **What is Dead Code Elimination?**
    Removing instructions that compute values never used later in the program.
69. **What is Temporary Variable Reduction?**
    Reusing temporary variables or eliminating them if their values are immediately consumed, reducing memory footprint.
70. **Why apply Local Optimization before Global Optimization?**
    Local optimization cleans up the basic blocks, reducing the size and complexity of the CFG for faster global analysis.
71. **What is loop unrolling?**
    Replicating the body of a loop to decrease the overhead of loop control instructions.
72. **What is loop invariant code motion?**
    Moving computations that yield the same result every iteration outside the loop.
73. **What is strength reduction?**
    Replacing expensive operations with cheaper ones (e.g., replacing `x * 2` with `x + x` or `x << 1`).
74. **What is peephole optimization?**
    A late-stage optimization examining a small sliding window (peephole) of machine instructions to replace them with faster equivalents.
75. **How does optimization affect debugging?**
    Optimized code is harder to debug because lines of code may be removed, reordered, or heavily modified.

## Section 7: General Compiler Design & Lab Prep
76. **What is Lex/Flex?**
    A tool for generating lexical analyzers (scanners) that use regular expressions to tokenise input.
77. **What is Yacc/Bison?**
    A tool for generating LALR parsers from context-free grammars.
78. **How do Flex and Bison communicate?**
    Flex returns tokens (integer IDs) to Bison via the `yylex()` function, and passes data via the `yylval` global variable.
79. **What does `%union` do in Bison?**
    It defines a C union allowing `yylval` to hold different data types for different tokens/non-terminals.
80. **What is the purpose of `yyparse()`?**
    It is the main entry point generated by Bison that initiates the parsing process.
81. **What is `yyerror()`?**
    A function called by the parser when a syntax error is encountered.
82. **What is the significance of `%left` and `%right`?**
    They define associativity and precedence for operators to resolve shift/reduce conflicts.
83. **What is a shift/reduce conflict?**
    When the parser cannot decide whether to shift the next token onto the stack or reduce the current stack items using a rule.
84. **What is a reduce/reduce conflict?**
    When the parser can reduce the stack using more than one grammar rule.
85. **Why do we use C to implement this compiler?**
    C provides low-level memory management and pointers, making it highly efficient for building linked lists and graphs.
86. **How do you compile a Flex/Bison project?**
    `flex file.l`, `bison -d file.y`, `gcc lex.yy.c parser.tab.c ...`
87. **What is the Symbol Table used for in this project?**
    Generating and tracking unique temporary variables and labels.
88. **What phase of the compiler comes after Intermediate Code Generation?**
    Code Optimization, followed by Target Code Generation.
89. **What is an Abstract Syntax Tree (AST)?**
    A tree representation of the syntactic structure of source code, often built before generating TAC.
90. **Why did we generate TAC directly without an AST?**
    This is called Syntax-Directed Translation. Since we only implemented one pass with backpatching, we generated TAC directly within the Bison actions.
91. **What is the main advantage of backpatching over a two-pass approach?**
    It saves parsing the file twice, making compilation faster and using less memory.
92. **How does Python's Tkinter relate to this compiler?**
    It acts merely as a Graphical User Interface to invoke the C executable and format the console output visually.
93. **What does `subprocess.run` do in the Python GUI?**
    It spawns a new process to run the `compiler.exe` and captures its standard output.
94. **Why is it important to check for `next_instr` continuously?**
    To accurately maintain the index of the next TAC instruction so jumps land correctly.
95. **What happens if a user writes a syntax error like `if a < ` without `b`?**
    Bison will fail to match the grammar rule and invoke `yyerror("syntax error")`.
96. **How does Dead Code Elimination work?**
    By checking if a computed temporary or variable is ever read again before being overwritten or exiting the scope.
97. **Can a Basic Block be empty?**
    No, a Basic Block must contain at least one instruction (its leader).
98. **Are comments included in Basic Blocks?**
    No, comments are stripped out during the lexical analysis phase by Flex.
99. **What makes your implementation suitable for academic submission?**
    It demonstrates all standard middle-end textbook algorithms: SDT, TAC, Backpatching, Block Partitioning, CFG formulation, and localized optimization.
100. **What is the most challenging part of implementing backpatching?**
     Correctly passing and merging the `truelist`, `falselist`, and `nextlist` up the parsing tree without losing track of pending jumps.
