## Execution Steps & GUI Guide

### Prerequisites
- **MSYS2 Environment** on Windows.
- **GCC**: Install using `pacman -S mingw-w64-x86_64-gcc`
- **Flex**: Install using `pacman -S flex`
- **Bison**: Install using `pacman -S bison`
- **Python 3**: For running the GUI (Tkinter is included by default).
- **Python Dependencies**: Install them by running:
  ```bash
  pip install -r requirements.txt
  ```


### Compilation (CLI)
1. Open MSYS2 MinGW64 terminal.
2. Navigate to the `Compiler_Project` directory.
3. Run the provided batch script or manually type the commands:
   ```bash
   flex lexer.l
   bison -d parser.y
   gcc -o compiler.exe lex.yy.c parser.tab.c tac.c backpatch.c basicblock.c cfg.c optimizer.c symboltable.c main.c
   ```
4. Verify `compiler.exe` is successfully generated.

### Running the CLI
To test the compiler on a source file:
```bash
./compiler.exe input.txt
```
This will print the TAC, Basic Blocks, CFG, and Optimized TAC to the console.

### Running the GUI
For a visual, user-friendly interface:
1. Ensure the project is compiled and `compiler.exe` exists in the folder.
2. Run the Python GUI script:
   ```bash
   python gui.py
   ```
3. **GUI Steps**:
   - Click **Load Source File** and select `input.txt` (or any custom code file).
   - The code will be displayed in the left pane.
   - Click **Run Compiler**.
   - The right pane contains multiple tabs:
     - **Original TAC**: Shows the unoptimized Three Address Code generated by the parser.
     - **Basic Blocks**: Shows the identified Leaders and instructions grouped into Basic Blocks.
     - **CFG (Text & DOT)**: Shows the predecessors/successors adjacency list and Graphviz DOT format.
     - **Optimized TAC**: Shows the intermediate code after applying Constant Folding and Common Subexpression Elimination.
     - **Raw Console Output**: The exact stdout generated by the C executable.

## Project Architecture

### Files & Modules
- **`lexer.l`**: Flex scanner definitions. Tokenizes the input, ignores whitespace, and passes IDs/Numbers to Bison.
- **`parser.y`**: Bison grammar file. Contains the syntactic rules for the language and embedded semantic actions for generating TAC and handling backpatching.
- **`tac.c` / `tac.h`**: Manages the generation of quadruples. Maintains a global linked list of TAC instructions.
- **`backpatch.c` / `backpatch.h`**: Implements `makelist`, `merge`, and `backpatch` for short-circuit boolean evaluation and control flow jump targets.
- **`symboltable.c` / `symboltable.h`**: Manages temporary variable generation (`t1`, `t2`) and label generation.
- **`basicblock.c` / `basicblock.h`**: Scans the TAC to identify Leaders (rules: first instruction, jump target, instruction after jump). Constructs Basic Blocks based on leaders.
- **`cfg.c` / `cfg.h`**: Connects basic blocks to form a Control Flow Graph. Generates adjacency lists and DOT notation.
- **`optimizer.c` / `optimizer.h`**: Applies local code optimizations (Common Subexpression Elimination and Constant Folding) within basic blocks.
- **`main.c`**: Entry point orchestrating parsing, basic block formation, CFG construction, and optimization passes.
- **`gui.py`**: Python Tkinter application providing a graphical front-end.

## Report Content

### Abstract
This project presents the design and implementation of a compiler front-end and middle-end focusing on Intermediate Code Generation (ICG). It translates structured control statements (Assignments, If-Else, While loops) into Three Address Code (TAC). Through the implementation of short-circuit evaluation using backpatching, the compiler accurately models boolean logic. Furthermore, it implements flow analysis by identifying Basic Blocks and constructing a Control Flow Graph (CFG), paving the way for Local Code Optimization techniques such as constant folding and common subexpression elimination.

### Introduction
Intermediate Code Generation is a critical phase in compiler design, bridging the gap between high-level language parsing and machine-code generation. By producing a machine-independent intermediate representation (like TAC), compilers can perform powerful optimizations. This case study focuses on building these core middle-end structures from scratch using C, Flex, and Bison.

### Problem Statement
To translate structured control statements containing boolean and arithmetic expressions into Three Address Code, handle jump resolution via backpatching, group instructions into Basic Blocks, map their execution flow via a Control Flow Graph, and apply local optimizations to reduce code redundancy.

### Objectives
1. Tokenize and parse assignments, conditionals, and loops.
2. Generate Three Address Code using Quadruples.
3. Implement short-circuit evaluation for boolean expressions.
4. Resolve incomplete jumps using backpatching.
5. Identify Leaders and construct Basic Blocks.
6. Build a CFG representing block successors and predecessors.
7. Perform local optimizations on the TAC.

### Intermediate Code Generation Design
Three Address Code (TAC) simplifies complex expressions into basic operations containing at most three operands. 
- Example: `x = (a + b) * c` becomes `t1 = a + b`, `t2 = t1 * c`, `x = t2`.
- **Quadruples**: Our implementation stores TAC as quadruples containing `op`, `arg1`, `arg2`, and `result`.

### Backpatching Design
In a one-pass compiler, jump targets for forward branches are unknown at the time the jump is generated. Backpatching solves this:
- **`makelist(i)`**: Creates a list containing instruction `i`.
- **`merge(p1, p2)`**: Concatenates two lists of instructions.
- **`backpatch(p, target)`**: Inserts the `target` instruction number into all instructions in list `p`.
This allows precise generation of short-circuit AND/OR logic without multiple passes.

### Basic Block & CFG Design
- **Leader Identification Rules**: 
  1. The first TAC instruction is a leader.
  2. Any target of a conditional or unconditional jump is a leader.
  3. Any instruction immediately following a jump is a leader.
- **Basic Blocks**: A sequence of consecutive TAC instructions starting with a leader and ending just before the next leader.
- **Control Flow Graph**: Nodes are Basic Blocks. Directed edges exist from B1 to B2 if control can flow from the end of B1 to the start of B2 (either via a jump or a natural fall-through).

### Optimization Design
Local optimization operates within the confines of a single Basic Block.
- **Constant Folding**: Evaluates expressions with known constants at compile time (e.g., `2 + 3` becomes `5`).
- **Common Subexpression Elimination**: Identifies if an expression `a + b` was previously computed and assigned to a temporary, replacing subsequent identical computations with a direct copy.

### Results & Advantages
The implemented compiler successfully reads structured code and outputs optimized TAC. 
- **Advantages**: Modular architecture; demonstrates complex compiler theory (backpatching) practically; easy to visualize flow through the provided GUI.
- **Limitations**: Optimization is currently local (intra-block) rather than global (inter-block).
- **Future Scope**: Implementation of global data-flow analysis, liveness analysis, and machine-code generation for a specific architecture.
